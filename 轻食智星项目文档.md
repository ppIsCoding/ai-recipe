## 一、项目背景

### 1.1 宏观政策与社会背景

随着"健康中国2030"战略深入推进，国民健康管理已上升为国家战略层面。《中国居民膳食指南（2022）》明确提出"合理膳食、健康生活"的核心理念，倡导居民建立科学的饮食习惯。与此同时，我国职场人群规模持续扩大，据国家统计局数据显示，2024年全国就业人员达7.3亿，其中城镇就业人员超4.7亿，这一庞大群体的健康状况直接关系到社会生产力与经济发展质量。

然而，快节奏的职场生活与健康管理之间存在天然矛盾。加班文化、高强度工作压力以及碎片化的生活方式，使得职场人群在饮食管理方面面临诸多现实困境。

### 1.2 职场人群饮食现状与痛点

**（1）饮食不规律现象普遍**

根据《2024中国职场人健康洞察报告》显示，76%的职场人存在不同程度的饮食不规律问题，主要表现为：早餐匆忙或省略、午餐依赖外卖、晚餐时间不固定或过度进食。长期不规律饮食导致胃肠疾病、肥胖、营养失衡等健康问题频发。

**（2）外卖依赖与健康隐患**

美团研究院数据显示，工作日午餐外卖订单量是周末的2.3倍，职场人群已成为外卖消费的主力军。然而，外卖食品普遍存在高油、高盐、高热量的问题，长期食用容易引发代谢综合征。丁香医生调查指出，68%的年轻人表示"知道应该健康饮食但难以坚持"，认知与行动之间存在显著落差。

**（3）决策疲劳与烹饪门槛**

"今天吃什么"已成为职场人每日面临的高频决策难题。下班后精力耗尽，面对冰箱中的零散食材，往往难以快速判断能做什么、怎么做。同时，烹饪技能的缺失进一步提高了下厨门槛——"切多大、先放什么、火候怎么控制"等问题成为阻碍职场人自己做饭的主要障碍。

**（4）食材浪费与心理负担**

由于缺乏有效的食材管理意识和能力，职场用户常出现买菜后忘记使用、剩余食材难以搭配、临期食材被浪费等情况。这不仅造成经济损失，还会产生负罪感和挫败感，形成"买菜—遗忘—浪费—自责"的恶性循环。

### 1.3 现有解决方案的局限性

当前市场上虽有部分健康管理或菜谱类应用，但针对职场人群饮食场景的解决方案仍存在明显不足：

| 现有产品类型 | 代表产品 | 局限性 |
|--------------|----------|--------|
| 健康管理类 | 薄荷健康、MyFitnessPal | 侧重热量记录，缺乏烹饪指导，操作繁琐 |
| 菜谱工具类 | 下厨房、美食杰 | 菜谱泛化，缺乏个性化推荐，无智能识别 |
| 健身饮食类 | Keep饮食 | 目标用户为健身人群，与职场人场景不匹配 |
| 外卖平台 | 美团、饿了么 | 加剧外卖依赖，无法解决健康饮食需求 |

上述产品普遍存在以下问题：**功能单一**（仅做记录或仅提供菜谱）、**场景泛化**（未针对职场人特定需求优化）、**交互繁琐**（需手动搜索、输入信息多）、**缺乏智能**（无法根据现有食材自动生成方案）。

### 1.4 技术发展带来的机遇

近年来，人工智能技术的快速发展为解决上述痛点提供了新的可能：

1. **多模态大模型成熟**：以通义千问、GPT-4V为代表的多模态模型具备强大的图像理解能力，可实现食材的自动识别与分类，大幅降低用户输入成本。

2. **大语言模型普及**：基于LLM的智能问答能力可为用户提供个性化的烹饪指导和营养建议，使"AI私人营养师"成为可能。

3. **小程序生态完善**：微信小程序凭借"轻量、社交、连接"的特性，成为承载高频、碎片化服务的理想载体。其即用即走的特点完美契合职场人"临时起意、快速决策"的使用场景。

4. **云计算成本下降**：AI模型调用成本持续降低，使得面向个人用户的智能化服务具备商业可行性。

## 二、设计目标

### （一）项目概述

本项目拟开发一款面向职场人群的 AI 健康膳食规划与烹饪助手——"轻食智星"（以下简称"轻食智星"）。项目以微信小程序为主要载体，融合图像识别、大语言模型、个性化推荐和营养评估等技术，针对职场人时间碎片化、下班后决策精力不足、既希望吃得更健康又不愿承担复杂烹饪成本等现实特征，提供从食材识别、膳食规划、菜谱推荐到烹饪指导和营养反馈的一体化服务，并在功能设计中同步回应其对确定感、掌控感与可持续生活秩序的现实需求。

### （二）项目定位

"轻食智星"立足职场人高频、碎片化的日常饮食场景，围绕**"以尽可能低的决策成本完成一餐安排"**的设计理念，聚焦**"吃什么、怎么做、是否健康"**三个核心问题，既降低用户的认知负担，也帮助其在有限时间内建立更稳定的生活掌控感。

与现有产品相比，"轻食智星"的核心差异化在于：

| 维度 | 现有产品 | 轻食智星 |
|------|----------|----------|
| 目标用户 | 泛人群或健身人群 | 职场人群（精准聚焦） |
| 核心场景 | 泛烹饪/泛健康 | 工作日晚餐（高频刚需） |
| 输入方式 | 手动搜索/选择 | 拍照识别（低门槛） |
| 智能程度 | 低（工具型） | 高（AI驱动） |
| 服务链路 | 单一功能 | 识别→推荐→烹饪→反馈（闭环） |

### （三）总体目标

项目总体目标是构建**"食材感知—智能推荐—烹饪执行—健康反馈"**的完整服务闭环。系统在职场用户时间有限、决策带宽不足的真实情境下，通过低输入方式识别现有食材、结合健康目标快速生成可执行方案，并在持续沉淀用户偏好、收藏和烹饪历史的基础上形成用户画像，在降低决策疲劳的同时，为用户提供更精准、更可坚持的膳食服务。

**核心价值主张**：以尽可能低的决策成本，帮助职场用户完成一餐安排。

### （四）具体目标

**1. 提升工作日膳食决策效率**

根据现有食材、用餐时间和口味偏好，推荐更适合下班后快速执行的餐食方案，缓解用户在疲劳状态下的选择压力。系统通过智能算法将"今天吃什么"这一开放性问题转化为具体的、可执行的选项，将决策时间从平均15-20分钟缩短至2-3分钟。

**2. 强化可坚持的健康饮食引导**

结合《中国居民膳食指南（2022）》、BMI、基础代谢和健康目标，在忙碌作息下提供更易执行的均衡饮食建议，缩小健康认知与实际行动之间的落差。系统不仅告诉用户"应该吃什么"，更提供"现在能吃什么"的具体方案。

> 图 1 膳食指南参考标准

**3. 降低下厨启动门槛**

以清晰步骤、语音播报和问答辅助帮助用户在精力有限时也能完成基础餐食，降低畏难感与失败预期。通过"防失败"的执行支撑设计，让用户从"不敢做"变为"做得成"。

**4. 减少食材浪费与重复采购**

优先利用已有食材进行推荐，提升食材利用率并降低因遗忘造成的浪费，同时减轻由此产生的负罪感和挫败感。系统帮助用户将"冰箱里的零散食材"转化为"可执行的一餐"。

**5. 构建持续学习的个性化服务平台**

通过前后端分离和模块化设计沉淀用户画像，为后续精准服务和功能扩展预留空间，并逐步增强用户的长期使用黏性与自我管理能力。随着使用次数增加，系统推荐将越来越贴合用户个人偏好。

## 三、应用场景

### 3.1 工作日晚餐规划

**场景描述**

职场用户下班后时间有限，决策精力也已被工作消耗，面对冰箱里零散食材时，往往难以快速判断能做什么、是否营养均衡，典型表现为决策带宽不足和行动启动困难。

**解决方案**

用户将现有食材拍照上传，系统识别食材后，根据口味偏好、烹饪时间和健康目标推荐合适菜谱，并给出清晰的制作步骤。用户不需要再进行复杂搭配和反复权衡，从而降低选择疲劳，并更容易把"想做饭"转化为"能做成"。

**典型用户画像**

- 张先生，28岁，互联网公司程序员，经常加班到晚上8点
- 冰箱里有鸡蛋、西红柿、西兰花，但不知道能做什么
- 使用"轻食智星"拍照识别后，系统推荐"西红柿炒蛋+蒜蓉西兰花"，预计20分钟完成

### 3.2 健康饮食与体重管理

**场景描述**

部分职场人长期依赖外卖、加班餐或高油高盐饮食，在作息不规律和久坐压力下，容易出现体重上升、营养结构单一和精力不足等问题，同时也普遍存在"知道不健康但难以持续改变"的心理张力。

**解决方案**

用户在系统中填写身高、体重、年龄、活动水平和目标状态，系统计算 BMI、BMR、TDEE 等指标，结合少油、少盐、高蛋白、均衡搭配等原则推荐餐食，并通过营养分析帮助用户在忙碌节奏中理解每餐的能量和营养构成，将健康焦虑转化为更可执行的选择。

**核心指标说明**

| 指标 | 含义 | 应用场景 |
|------|------|----------|
| BMI | 身体质量指数 | 判断体重是否在健康范围 |
| BMR | 基础代谢率 | 估算静息状态下的能量消耗 |
| TDEE | 每日总能量消耗 | 确定每日热量摄入目标 |

### 3.3 烹饪基础薄弱人群入门

**场景描述**

青年职场人、独居白领或刚开始独立生活的用户，往往缺乏基础烹饪经验，面对"切多大、先放什么、火候怎么控制"等问题容易放弃做饭，其本质是担心失败、担心麻烦以及担心投入与结果不成正比。

**解决方案**

系统在菜谱详情中提供食材准备、处理方法、烹饪步骤和注意事项。用户进入烹饪指导页面后，可按照当前步骤逐项完成操作，也可以通过语音播报减少查看手机的频率。遇到疑问时，可直接向智能助手提问，获得更适合当前生活节奏的通俗建议，从而形成一种"防失败"的执行支撑。

**功能特色**

- **分步骤引导**：每一步只显示当前需要完成的操作，避免信息过载
- **语音播报**：解放双手双眼，专注于烹饪操作
- **智能问答**：随时提问，获得通俗易懂的解答

### 3.4 食材管理与减少浪费

**场景描述**

职场用户常因工作繁忙出现买菜后忘记使用、剩余食材难以搭配、临期食材被浪费等情况，这类浪费不仅带来成本损失，也会形成明显的心理负担。

**解决方案**

系统优先根据已有食材推荐菜谱，并在推荐时考虑食材类别和搭配关系，帮助用户在有限时间内把手头资源转化为可执行的一餐，从而减少浪费并缓解由浪费带来的负面情绪。

**预期效果**

- 食材利用率提升30%以上
- 减少因遗忘导致的食材浪费
- 降低"买菜焦虑"和"浪费负罪感"

### 3.5 特殊饮食需求

**场景描述**

不同职场用户可能存在忌口、过敏、控糖、增肌、减脂、素食等个性化需求，通用菜谱工具往往难以兼顾效率与适配度。

**解决方案**

用户可设置口味偏好、忌口食材和健康目标。系统在推荐菜谱时自动规避不适合的食材，并优先匹配更符合用户目标的餐食类型，提高推荐结果的安全性、适配性与实际执行价值。

**支持的个性化设置**

| 设置类型 | 示例选项 |
|----------|----------|
| 口味偏好 | 清淡、微辣、重辣、酸甜等 |
| 忌口食材 | 海鲜、香菜、葱姜蒜等 |
| 健康目标 | 减脂、增肌、控糖、均衡饮食 |
| 饮食类型 | 素食、低碳水、高蛋白等 |

## 四、采用的技术

### 4.1 技术架构概述

"轻食智星"采用前后端分离架构，整体技术组织遵循**"前端轻量交互 + 后端统一调度 + 模型能力组合 + 数据持续沉淀"**的设计原则。

> 图 2 系统技术架构图

| 技术层 | 职责 | 核心技术 |
|--------|------|----------|
| 前端交互层 | 用户交互、图片上传、结果展示 | 微信小程序原生开发 |
| 服务接口层 | 业务逻辑、数据调度 | FastAPI + Uvicorn |
| 数据存储层 | 用户档案、菜谱数据、行为记录 | MySQL 8.0 + SQLAlchemy |
| 模型智能层 | 图像识别、问答、营养分析 | 通义千问多模态模型 + 通义千问大语言模型 |
| 工作流编排层 | 任务调度、流程控制 | LangGraph Agent |

### 4.2 前端技术

采用**微信小程序原生开发**，充分利用小程序"轻量、社交、连接"的生态特点。系统包含11个页面，采用底部三Tab导航结构（首页、AI助手、我的）。

**核心能力集成**

| 能力 | 应用场景 |
|------|----------|
| 拍照/相册选择 | 食材图片上传 |
| 本地缓存 | 用户偏好、历史记录离线存储 |
| 语音合成（同声传译插件） | 烹饪步骤语音播报 |
| 微信登录 | 用户身份认证 |
| 消息订阅 | 做饭提醒、健康提醒 |
| 分享能力 | 菜谱分享到朋友圈/群聊 |

### 4.3 后端技术

采用 **FastAPI** 框架构建后端接口服务，主要技术组件：

| 组件 | 用途 |
|------|------|
| FastAPI + Uvicorn | Web框架与ASGI服务器 |
| SQLAlchemy | ORM框架，数据库访问 |
| LangGraph | 工作流编排，构建智能Agent |
| httpx | 异步HTTP客户端，调用AI服务 |

系统提供7个核心API接口：食材识别、菜谱推荐、智能问答、用户管理、收藏管理、做菜历史、营养分析。

### 4.4 数据库设计

采用 **MySQL 8.0** 作为主数据库，通过 SQLAlchemy ORM 进行数据访问。数据库包含用户表、菜谱表、收藏表、做菜历史表和营养分析缓存表，分别存储用户基础信息与健康档案、菜谱数据与营养成分、用户收藏记录、烹饪历史以及营养分析结果。

**健康标签体系**（基于《中国居民膳食指南2022》）

系统设计了少盐、少油、高蛋白、轻烹饪四类健康标签，用于辅助菜谱推荐与健康评分。

### 4.5 AI 能力集成

系统通过 **4 个 LangGraph Agent** 组织 AI 能力，每个 Agent 负责一个独立的智能任务：

| Agent | 功能 | 处理流程 |
|-------|------|----------|
| `recognition_agent` | 食材识别 | 多模态识别 → 结果格式化 |
| `recipe_agent` | 菜谱推荐 | 条件筛选 → 食材匹配 → 扩展搜索 → AI排序 |
| `chat_agent` | 智能问答 | 意图识别 → 回复生成 |
| `nutrition_agent` | 营养分析 | 营养查询 → 总量计算 → 建议生成 → 做法推荐 |

**AI服务封装**

系统通过 `QwenService` 类封装通义千问API调用，支持通用对话、多模态图片识别、流式对话和JSON格式返回4种调用方式。

**模型配置**
- 文本对话：通义千问大语言模型
- 图片识别：qwen3.5-omni-flash（多模态模型）
- 流式传输：支持SSE（Server-Sent Events）

### 4.6 稳定性设计

**缓存策略**

| 缓存层 | 缓存内容 | 策略 |
|--------|----------|------|
| 菜谱缓存 | 常用菜谱数据 | 启动时加载至内存 |
| 营养缓存 | 相同食材组合分析结果 | 数据库缓存，自动去重 |
| 会话缓存 | 对话上下文 | 保留最近10轮 |

**异常兜底**

| 异常场景 | 兜底策略 |
|----------|----------|
| 食材识别失败 | 提示重新拍照或手动输入 |

## 六、核心算法与原理

### 6.1 系统整体逻辑

"轻食智星"采用前后端分离架构，整体逻辑围绕职场用户"低输入、快决策、可执行"的使用特点进行组织。系统通过 4 个 LangGraph Agent 组织 AI 能力，每个 Agent 负责一个独立的智能任务，通过状态图（StateGraph）定义处理流程。

> 图 3 系统算法流程图

### 6.2 食材识别原理

食材识别模块以用户上传的图片为输入，通过通义千问多模态大模型（qwen3.5-omni-flash）判断图片中包含的常见食材，并输出结构化结果。

**识别流程**

```
用户上传图片 → Base64编码 → 多模态模型推理 → JSON结果解析 → 结构化食材列表
```

**输出字段说明**

| 字段 | 类型 | 说明 |
|------|------|------|
| name | string | 食材名称（如：鸡蛋、西红柿） |
| category | string | 分类（蔬菜/水果/肉类/海鲜/蛋奶/主食/调料/其他） |
| shelf_life_days | int | 冰箱建议保存天数 |
| probability | float | 识别置信度（0-1） |

**识别优化策略**

- **原材料优先**：只识别原材料和生鲜食材，不识别成品菜
- **成品菜逆推**：如果图片中是成品菜，尝试识别其原材料
- **常见名称**：使用常见的中文食材名称，避免学名

### 6.3 菜谱推荐算法

菜谱推荐算法围绕"可做性、适配性、健康性和可执行性"四项原则展开，采用"预筛选—匹配—扩展—排序"四阶段处理路径。

**算法流程**

```
输入食材 → 条件预筛选 → 食材匹配 → 匹配质量判断
                                          ↓           ↓
                                     质量好      质量差
                                          ↓           ↓
                                     AI排序    扩展搜索
                                          ↑           ↓
                                          └───────────┘
                                          ↓
                                     格式化输出 → END
```

**（1）条件预筛选**

系统首先依据最大烹饪时长、健康标签进行预筛选，优先保留更适合日常执行的轻烹饪方案（≤30分钟）。

**（2）食材匹配评分**

在候选池中，算法对用户食材与菜谱主料执行双向子串匹配：

```python
# 匹配逻辑示例
for food_name in foods:
    for ingredient in recipe["main_ingredients"]:
        if food_name in ingredient or ingredient in food_name:
            score += 1
            matched.append(ingredient)
```

**（3）匹配分档与扩展检索**

系统根据命中数量对匹配质量划分为三档：

| 档位 | 命中数量 | 处理策略 |
|------|----------|----------|
| good | ≥3 | 直接进入AI排序 |
| medium | 1-2 | 进入扩展搜索 |
| poor | 0 | 进入扩展搜索 |

扩展搜索策略：
- **策略1**：AI生成相似食材并扩展匹配
- **策略2**：同分类菜谱补足
- **策略3**：默认轻烹饪方案兜底

**（4）综合排序与理由生成**

进入最终候选集后，系统综合健康标签、用户目标、口味偏好、忌口设置进行排序，并生成简洁推荐理由。

**核心评价维度**

| 评价维度 | 关键依据 | 作用说明 |
|----------|----------|----------|
| 食材匹配度 | 用户已有食材与菜谱主料的重合程度 | 保证推荐结果具备实际可烹饪性 |
| 健康适配度 | 少油、少盐、高蛋白等健康标签 | 提升餐食质量与健康目标一致性 |
| 个性化程度 | 口味偏好、忌口设置 | 提高推荐接受度 |
| 执行难度 | 步骤数量、预计耗时 | 降低新手失败率 |
| 目标匹配度 | 减脂、维持、增肌等用户目标 | 增强长期价值 |

### 6.4 健康评分算法

系统根据用户健康档案和菜谱健康标签计算综合评分（0-100分）：

**评分规则**

| 评分项 | 分值 | 说明 |
|--------|------|------|
| 基础分 | 50 | 所有菜谱基础分 |
| 少盐标签 | +10 | 符合膳食指南准则5 |
| 少油标签 | +10 | 符合膳食指南准则5 |
| 高蛋白标签 | +5 | 符合膳食指南准则4 |
| 轻烹饪标签 | +5 | 符合职场场景需求 |
| 荤素搭配 | +10 | 符合膳食指南准则1 |

**用户目标调整**

| 用户目标 | 调整规则 |
|----------|----------|
| 减脂（lose_weight） | 少油标签+5，热量≤300kcal+5 |
| 增肌（gain_muscle） | 高蛋白标签+10 |
| 超重用户（BMI≥24） | 热量>400kcal扣10分 |

### 6.5 健康分析算法

健康分析模块以用户基础信息为输入，围绕 BMI、BMR、TDEE、目标热量和营养素目标等关键指标构建健康画像。

**（1）BMI 状态评估**

```
BMI = 体重(kg) / 身高(m)²
```

| BMI范围 | 状态 | 中国标准 |
|---------|------|----------|
| <18.5 | 偏瘦 | 偏瘦 |
| 18.5-24 | 正常 | 正常 |
| 24-28 | 超重 | 超重 |
| ≥28 | 肥胖 | 肥胖 |

**（2）BMR 计算（Mifflin-St Jeor 公式）**

```
男性：BMR = 10 × 体重(kg) + 6.25 × 身高(cm) - 5 × 年龄 - 161 + 166
女性：BMR = 10 × 体重(kg) + 6.25 × 身高(cm) - 5 × 年龄 - 161
```

**（3）TDEE 计算**

```
TDEE = BMR × 活动系数
```

| 活动水平 | 活动系数 |
|----------|----------|
| 久坐（sedentary） | 1.2 |
| 轻度活动（light） | 1.375 |
| 中度活动（moderate） | 1.55 |
| 重度活动（active） | 1.725 |

**（4）目标热量修正**

| 健康目标 | 热量调整 |
|----------|----------|
| 减脂（lose_weight） | TDEE - 500 |
| 维持（maintain） | TDEE |
| 增肌（gain_muscle） | TDEE + 300 |

**（5）营养素目标分配**

基于目标热量，按照膳食指南建议分配营养素：

| 营养素 | 供能比例 | 计算公式 |
|--------|----------|----------|
| 蛋白质 | 15-20% | 目标热量 × 0.175 / 4 |
| 脂肪 | 20-30% | 目标热量 × 0.25 / 9 |
| 碳水化合物 | 50-65% | 目标热量 × 0.575 / 4 |

### 6.6 智能问答与上下文增强

智能问答模块采用"意图识别 + 策略映射 + 上下文增强"的处理方式。

**（1）意图识别**

系统将用户问题识别为以下类别：

| 意图类别 | 触发场景 | 回复策略 |
|----------|----------|----------|
| recommend | 想要推荐菜谱、今晚吃什么 | 候选菜谱+推荐理由 |
| recipe | 询问具体菜谱做法 | 步骤、火候、要点 |
| nutrition | 营养健康问题 | 营养数据+科学解释 |
| health | 健康饮食建议 | 引用BMI、TDEE、膳食指南 |
| skill | 烹饪技巧、窍门 | 简洁实用的技巧说明 |
| chat | 日常闲聊 | 友好回复，引导饮食话题 |

**（2）上下文增强**

系统自动注入以下上下文信息：

- **用户健康档案**：BMI、TDEE、健康目标
- **用户偏好**：口味偏好、忌口设置
- **行为记录**：收藏记录、做菜历史
- **对话历史**：最近10轮对话

### 6.7 营养分析算法

菜品营养分析模块聚焦单次食材组合的即时反馈，采用"分析 → 计算 → 建议 → 推荐"四阶段处理。

**处理流程**

```
食材列表 → 营养成分查询 → 总量计算 → 健康建议生成 → 做法推荐
```

**输出内容**

| 输出项 | 说明 |
|--------|------|
| 营养成分 | 热量、蛋白质、脂肪、碳水、膳食纤维 |
| 健康建议 | 营养评价、适合人群、搭配建议 |
| 推荐做法 | 2-3种适合当前食材的简单做法 |

**缓存机制**

系统对相同食材组合的分析结果进行缓存，通过食材列表的哈希值（foods_key）进行去重，避免重复调用AI模型。

**（1）小程序前端实现**

采用微信小程序原生开发方式，完成以下核心页面：

- 首页：拍照识别入口、推荐菜谱展示
- 拍照页：食材拍照上传、相册选择
- 识别结果页：食材列表展示、勾选修正
- 菜谱详情页：食材清单、步骤引导、营养信息
- AI助手页：智能问答、流式对话
- 个人中心：健康档案、收藏、历史

**（2）后端服务实现**

采用 FastAPI 构建接口服务，主要模块：

- `main.py`：应用入口，生命周期管理，路由注册
- `api/`：API路由模块（recognize、recipe、chat、users、favorites、history、nutrition）
- `agents/`：LangGraph Agent模块（recognition、recipe、chat、nutrition）
- `services/`：外部服务封装（qwen_service）
- `models/`：数据库模型定义

**（3）AI 能力与工作流实现**

通过 4 个 LangGraph Agent 组织 AI 能力：

- `recognition_agent`：食材识别（多模态识别 → 结果格式化）
- `recipe_agent`：菜谱推荐（筛选 → 匹配 → 扩展 → 排序 → 格式化）
- `chat_agent`：智能问答（意图识别 → 回复生成）
- `nutrition_agent`：营养分析（营养查询 → 总量计算 → 建议生成 → 做法推荐）

**（4）数据库实现**

采用 MySQL 8.0 存储数据，SQLAlchemy ORM 进行数据访问：

- `users`表：用户基础信息、健康档案、口味偏好
- `recipes`表：菜谱数据、健康标签、营养成分
- `favorites`表：收藏记录
- `cook_history`表：做菜历史
- `nutrition_logs`表：营养分析缓存

### 7.3 优化措施

**（1）交互优化**

- 减少冗余输入，核心流程保持"上传食材—查看推荐—开始烹饪"的简单路径
- 复杂设置集中在个人中心，不干扰主流程
- 拍照识别替代手动搜索，降低输入负担

**（2）性能优化**

- 图片压缩：上传前压缩图片，减少传输时间
- 菜谱缓存：启动时加载至内存，减少数据库查询
- 营养缓存：相同食材组合的分析结果缓存，避免重复调用AI
- 异步接口：FastAPI异步处理，提升并发能力
- 数据库连接池：复用连接，减少连接开销

**（3）稳定性优化**

- 菜谱数据兜底：数据库加载失败时使用内置默认数据
- 推荐结果兜底：匹配不足时扩展搜索，默认轻烹饪方案
- AI调用兜底：模型调用超时时返回基于规则的备用推荐
- JSON解析兜底：正则提取JSON内容，解析失败返回空数组

**（4）数据优化**

- 收藏记录：联合唯一约束，避免重复收藏
- 做菜历史：索引优化（user_id、created_at），提升查询效率
- 营养缓存：foods_key唯一约束，自动去重

### 10.2 技术创新

**（1）多模态AI融合**

将通义千问多模态模型的图像理解能力与大语言模型的自然语言处理能力相结合，实现"拍照识别→智能推荐→自然语言问答"的全链路智能化。

**（2）LangGraph Agent 架构**

采用 LangGraph 工作流编排框架，将复杂的AI任务拆分为独立的Agent，通过状态图定义处理流程，实现：

- 模块化：每个Agent独立开发、测试、优化
- 可扩展：新增功能只需添加新的Agent
- 可维护：清晰的流程定义，便于调试和优化

**（3）健康评分算法**

基于《中国居民膳食指南2022》设计健康评分算法，综合考虑：

- 健康标签（少盐、少油、高蛋白、轻烹饪）
- 荤素搭配
- 用户个人健康档案（BMI、健康目标）
- 烹饪复杂度

实现个性化的健康菜谱推荐，而非简单的关键词匹配。

**（4）上下文增强问答**

智能问答模块自动注入用户健康档案、口味偏好、忌口设置、收藏记录、做菜历史和对话上下文，实现个性化的问答体验，避免用户重复说明。

### 10.3 产品创新

**（1）决策成本最小化**

- 拍照识别替代手动搜索，输入成本降至最低
- 三步完成核心流程（拍照→确认→推荐）
- 首次使用无需填写信息，即拍即用

**（2）防失败设计**

- 分步骤引导，每一步只显示当前操作
- 语音播报，解放双眼双手
- 智能问答兜底，随时提问获得帮助

**（3）健康认知转化**

- 将抽象的健康目标转化为具体的菜谱推荐
- 健康标签可视化，让用户直观了解每道菜的健康价值
- 营养分析即时反馈，帮助用户建立科学饮食认知

## 十二、其他说明

### 12.1 数据安全与隐私保护

系统仅收集实现功能所需的必要信息，例如用户登录标识、基础健康档案、口味偏好、收藏记录和做菜历史。相关数据主要用于个性化推荐、营养分析和用户画像构建，不采集与功能无关的敏感信息。

在接口访问和数据管理方面，系统采用统一后端入口进行数据读写控制，并将分析缓存、行为记录和个人信息分层管理，以降低无关访问和不必要暴露的风险。

**数据安全措施**

| 措施 | 说明 |
|------|------|
| 微信登录 | 使用微信官方登录机制，不存储用户密码 |
| API密钥管理 | 密钥通过环境变量配置，不硬编码在代码中 |
| 数据隔离 | 用户数据通过openid隔离，互不干扰 |
| 敏感信息脱敏 | 日志中不记录敏感信息 |

### 12.2 局限性与改进方向

**当前局限性**

| 局限 | 说明 |
|------|------|
| 问答不支持图片 | 智能问答模块暂不支持图片上传，无法在问答中直接识别食材 |
| 菜谱数据有限 | 当前预置菜谱数量有限，覆盖的菜系和口味不够全面 |
| 离线能力不足 | 核心功能依赖网络，离线场景下可用性有限 |

**改进方向**

| 方向 | 说明 |
|------|------|
| 多模态问答 | 引入多模态模型能力，支持问答中上传图片 |
| 菜谱数据扩充 | 持续扩充菜谱数据，覆盖更多菜系和口味 |
| 离线缓存增强 | 增强离线缓存能力，支持常用菜谱离线查看 |
| 语音交互 | 支持语音输入，进一步降低操作门槛 |
| 社区功能 | 增加用户分享、评论、点赞等社交功能 |

### 12.3 项目总结

"轻食智星"围绕职场人群饮食管理需求，将食材识别、智能推荐、烹饪执行、健康分析和行为沉淀有机结合，形成从"识别现有条件"到"生成可执行方案"再到"持续优化服务"的完整闭环。

项目的核心设计理念并非提供通用菜谱库，而是在时间碎片化、决策带宽有限的真实生活环境中，为职场用户提供更省心、更可执行、更贴近个体状态的智能膳食支持，并在长期使用中帮助其逐步建立更稳定的生活秩序感。

通过微信小程序的"轻量、社交、连接"生态特性，结合多模态AI、LangGraph工作流编排、健康评分算法等技术创新，"轻食智星"实现了低门槛、快决策、防失败的用户体验，为职场人群的健康饮食管理提供了一种新的解决方案。
